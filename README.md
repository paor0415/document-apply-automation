# 書類申請 自動化ツール

Webフォームへの書類申請フローを自動化する、Playwright + Tkinter のサンプルツールです。
UIに申請内容を入力して「申請を実行」を押すと、以下のブラウザ操作が自動で実行されます。

1. ポータルサイト(OOOサイト)にアクセス
2. ポータル上のリンクをクリックして書類申請サイトへ遷移
3. フォームに入力内容を自動入力
4. 添付ファイルをアップロード
5. 申請ボタンを押して送信

現在は**ダミーサイトに対して一連のフローが動作することを確認済み**の状態です。
実サイトへ接続するには [`config.py`](config.py) の設定値を差し替えてください
（→ [実サイトへの接続手順](#実サイトへの接続手順)）。

## 使用ライブラリ

| ライブラリ | 用途 | 備考 |
|---|---|---|
| **Playwright** | ブラウザの自動操作 | 唯一の外部依存。CDP接続機能もこれに含まれる |
| tkinter | 入力画面(UI) | Python標準ライブラリ。追加インストール不要 |
| http.server | ダミーサイトの配信 | Python標準ライブラリ |
| logging | ログ出力 | Python標準ライブラリ |

**外部依存は Playwright のみ**です。

### なぜUIにTkinterを選んだか

| | Tkinter（採用） | Streamlit | FastAPI + HTML |
|---|---|---|---|
| 追加インストール | 不要（標準ライブラリ） | 必要 | 必要 |
| Playwright同期APIとの相性 | ◎ 素直に動く | △ 再実行モデルとイベントループに癖がある | ○ |
| 配布のしやすさ | ◎ Pythonがあれば動く | ○ | △ |
| 将来のWeb化 | △ | ◎ | ◎ |

**このツールはローカルChromeへCDP接続する都合上、利用者のPC上でしか動きません。**
そのためWebサーバー形式(Streamlit)の「どこからでもアクセスできる」という利点が活きません。
一方でPlaywrightの同期APIとの相性はTkinterの方が良いため、Tkinterを採用しました。

ただし [`ui.py`](ui.py) は `workflow.run_application_flow(data)` を呼ぶだけで、
ブラウザ操作のロジックを一切持っていません。将来Web化する場合も
`ApplicationData` を組み立てて同じ関数を呼ぶだけで済み、**他のファイルは変更不要**です。

## セットアップ

```bash
# 1. ライブラリをインストール（仮想環境を使う場合は事前に有効化しておく）
pip install -r requirements.txt

# 2. ブラウザ本体をインストール（これを忘れると起動に失敗します）
python -m playwright install chromium
```

## 使い方

### UIを起動する（通常はこれ）

```bash
python main.py
```

入力画面が立ち上がります。氏名・申請日・部署・書類名・申請理由を入力し、
「参照...」から添付ファイルを選んで「申請を実行」を押すと、フローが自動実行されます。

**ブラウザは画面に表示されません**（バックグラウンドで最後まで自動実行されます）。
何が行われたかは、UIの「**処理フロー**」欄で確認します:

- 各ステップ（接続 → ポータル → 遷移 → 入力 → 添付 → 送信）の状態がリアルタイムに更新される
  （○未実行 / ▶実行中 / ✅成功 / ⏭スキップ / ❌失敗）
- **ステップをクリックすると詳細ウィンドウが開き**、そのステップで実施された処理の内容
  （アクセスしたURL、入力した値、サイトからの応答など）と、実行時のブラウザ画面の
  **証跡スクリーンショット**を確認できる
- 証跡画像には「🤖 今なにをした段階か」のバナーと、入力済みの欄の緑枠が焼き込まれている

> 初期設定では**ダミーサイト**に対して実行されます。ダミーサーバーは `main.py` が自動起動します。
> ブラウザの動きを直接目視したい場合は `config.py` の `HEADLESS = False` にしてください。

### UIなしで動作確認する

```bash
python main.py --demo            # ダミーデータで一連のフローを実行（ブラウザが表示される）
python main.py --demo --headless # ブラウザを表示せずに実行
```

### その他のオプション

| コマンド | 説明 |
|---|---|
| `python main.py --serve` | ダミーサーバーだけ起動（ブラウザで手動確認したいとき） |
| `python main.py --cdp` | CDP接続を強制する（`config.USE_CDP` より優先） |
| `python main.py --no-cdp` | ブラウザ新規起動を強制する |
| `python main.py --debug` | 詳細ログを出力する |

### ブラウザの動きを直接目視したい場合

既定ではブラウザは画面に出ません（`config.py` の `HEADLESS = True`）。動きを直接見たい場合は
`HEADLESS = False` にしてください。このとき次の設定が効きます:

- `KEEP_BROWSER_OPEN = True`（既定）: フロー完了後もブラウザを閉じずに残す。確認後、✕ボタンで手動で閉じる
- `SLOW_MO_MS`: 1操作ごとに待つミリ秒。`800`〜`3000` にすると操作を1つずつ目視できる（既定は 0）
- `STEP_PAUSE_MS`: 節目（ページを開いた・入力が終わった等）ごとの小休止ミリ秒（既定は 0）

> `python main.py --demo` はUIを介さずスクリプト自体がすぐ終了するため、`KEEP_BROWSER_OPEN=True` でも
> スクリプト終了と同時にブラウザも一緒に終了します（Playwrightの仕組み上の制約）。ブラウザを開いたまま
> 確認したい場合は、`--demo` ではなく通常の `python main.py`（UI）を使ってください。

## 実サイトへの接続手順

差し替えが必要なのは **[`config.py`](config.py) と [`ui.py`](ui.py) の一部だけ**です。
該当箇所にはすべて `【要差し替え】` とコメントを入れてあります。

```bash
# 差し替え箇所を一覧で確認する
grep -n "要差し替え" *.py
```

### 手順1: 起動済みChromeにデバッグポートを開く

対象サイトがSSOやクライアント証明書認証を使っている場合、Playwrightが新規起動した
まっさらなブラウザではログインできません。そこで**普段使っているChromeを
デバッグポート付きで起動し、そこに外から接続**します。認証はChrome側で済ませます。

```bash
# macOS（※既存のChromeをすべて終了してから実行してください）
"/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" \
  --remote-debugging-port=9222 \
  --user-data-dir="$HOME/chrome-debug-profile"
```

```bat
REM Windows
"C:\Program Files\Google\Chrome\Application\chrome.exe" ^
  --remote-debugging-port=9222 ^
  --user-data-dir="%USERPROFILE%\chrome-debug-profile"
```

起動したChromeで**ポータルサイトに手動でログインしておきます**。
`http://localhost:9222/json/version` にアクセスしてJSONが表示されれば、ポートが開いています。

> `--user-data-dir` に既存と別のフォルダを指定しているため、初回はログインし直しになります。
> 2回目以降はこのプロファイルにセッションが残ります。

### 手順2: config.py を書き換える

| 設定項目 | 変更内容 |
|---|---|
| `USE_CDP` | `True` にする（起動済みChromeへ接続する） |
| `START_DUMMY_SERVER` | `False` にする（ダミーサーバーを起動しない） |
| `PORTAL_URL` | 実際のポータルサイトURL |
| `APPLY_URL_PREFIX` | 実際の申請ページURLの先頭部分 |
| `PortalSelectors.apply_link` | ポータル上の「書類申請」リンクのセレクタ |
| `FormSelectors` の各項目 | 申請フォームの各要素のセレクタ |

### 手順3: セレクタを調べて差し替える

1. Chromeで対象ページを開き **F12**（開発者ツール）
2. 対象の要素を右クリック →「**検証**」
3. ハイライトされた要素を右クリック →「**Copy**」→「**Copy selector**」
4. コピーした値を `config.py` の該当項目に貼り付け

**セレクタ選びのコツ:**

- 実サイトでは `id="mat-input-3"` のようにidが自動生成されることがあります。
  この場合idは画面を開くたびに変わる可能性があるため使わず、`name` 属性やラベル文言で指定してください。
- CSSセレクタ以外に、Playwrightの以下の記法もそのまま使えます:

  ```python
  submit_button: str = "text=申請する"                # 表示テキストで指定
  submit_button: str = "role=button[name='申請する']"  # アクセシビリティroleで指定
  submit_button: str = "xpath=//button[@type='submit']"  # XPathで指定
  ```

- **添付ファイル欄は、見た目の「参照」ボタンではなく `input[type=file]` 要素を指定してください。**
  `display:none` で隠されていても動作します。開発者ツールのコンソールで
  `document.querySelectorAll('input[type=file]')` を実行すると対象を探せます。

### 手順4: 部署の選択肢を合わせる（[`ui.py`](ui.py)）

```python
# ui.py の冒頭
DEPARTMENTS: list[tuple[str, str]] = [
    ("開発部", "dev"),      # (画面表示, フォームに渡す値)
    ...
]
```

実サイトの `<select>` の `value` 属性が不明な場合は、右側に表示ラベル（例: `"開発部"`）を
入れても動きます。`form_filler._select_option()` が value属性・表示ラベルの
**どちらでも選択できる**ようになっているためです。

### 手順5: DRY_RUN で安全に試す

いきなり本番送信すると事故につながるため、まず `config.py` で `DRY_RUN = True` にしてください。
**送信ボタンを押す直前で停止**し、そこまでの動作を確認できます。
`logs/screenshots/05_dry_run_stopped.png` で入力後のフォームを目視確認し、
問題なければ `DRY_RUN = False` に戻して本番送信します。

## 設計の要点

### 差し替え箇所を config.py に集約

セレクタ・URL・タイムアウトはすべて `config.py` に定義し、
ブラウザ操作のコード側にはハードコードしていません。
実サイト適用時に触るファイルを最小化するためです。

### 要素の型を自動判別

実サイトのフォームは、同じ「部署」欄でも `<select>` かもしれないし `<input>` かもしれません。
`form_filler._fill_field()` は要素のタグ種別を実行時に判別して入力方法を切り替えるため、
**セレクタを変えるだけで**大抵のフォームに対応できます。

### 新規タブ・同一タブの両対応

ポータルのリンクが `target="_blank"` で新規タブを開くか、同一タブ内で遷移するかは
実サイトを見るまで分かりません。`click_link_to_apply()` はまず新規タブの出現を待ち、
現れなければ同一タブ遷移とみなすため、**どちらでも動きます**。

### 利用者のChromeを壊さない

CDP接続時、`close()` は接続を切るだけで**利用者のChromeは終了しません**。
また、既に開いているタブを乗っ取らないよう、常に新しいタブを開いて作業します。

### 失敗の種類ごとに例外を分ける

`exceptions.py` で失敗を種類ごとに分けており、それぞれ**原因と対処法**をメッセージに含めています。

| 例外 | 発生する状況 |
|---|---|
| `BrowserConnectionError` | Chromeに接続できない／起動できない |
| `NavigationError` | サイトにアクセスできない／想定と違うページに遷移した |
| `ElementNotFoundError` | セレクタの要素が見つからない（セレクタ誤り or サイト変更） |
| `FileAttachError` | 添付ファイルが無い／空／input要素が見つからない |
| `SubmitError` | 送信ボタンが無い・押せない／サイトが申請を却下した |
| `ValidationError` | 入力値の不備（ブラウザを開く前に検出） |

## ログとスクリーンショット

### ログ

- **画面**: UI下部の「実行ログ」欄にリアルタイム表示（警告=橙、エラー=赤）
- **ファイル**: `logs/automation.log`（日次ローテーション・30日保持）
- **コンソール**: 標準出力

### スクリーンショット

各ステップで `logs/screenshots/` に自動保存されます。**失敗時の原因調査に有用です。**

| ファイル | 内容 |
|---|---|
| `01_portal.png` | ポータルサイトを開いた直後 |
| `02_apply_form.png` | 申請サイトへ遷移した直後 |
| `03_form_filled.png` | フォーム入力後 |
| `04_file_attached.png` | ファイル添付後 |
| `05_result.png` | 送信結果 |
| `05_submit_timeout.png` | 結果画面を確認できなかった場合 |

不要な場合は `config.py` の `SAVE_SCREENSHOTS = False` で無効化できます。

## トラブルシューティング

| 症状 | 原因と対処 |
|---|---|
| `Executable doesn't exist` | ブラウザ本体が未導入。`python -m playwright install chromium` を実行 |
| `起動済みChromeへの接続に失敗しました` | デバッグポートが開いていない。**既存のChromeをすべて終了**してから、`--remote-debugging-port=9222` 付きで起動し直す |
| `〜が見つかりませんでした（セレクタ: ...）` | `config.py` のセレクタが実サイトと不一致。開発者ツールで正しいセレクタを確認 |
| `想定と異なるページに遷移しました` | ログイン画面へリダイレクトされている可能性。接続先Chromeでログイン状態を確認 |
| `〜に N 個の要素が該当してしまいました` | セレクタが緩すぎる。より具体的な指定に変更 |
| `申請ボタンが押せない状態（無効）です` | 必須項目が未入力の可能性。`logs/screenshots/03_form_filled.png` を確認 |
| `送信後、結果画面を確認できませんでした` | **申請自体は完了している可能性あり**。実サイトで申請状況を確認したうえで、`success_indicator` を修正 |
| サイトが重くてタイムアウトする | `config.py` の `ELEMENT_TIMEOUT_MS` / `NAVIGATION_TIMEOUT_MS` を延ばす |
| 動作を1つずつ目視したい（実サイトのセレクタ調整時など） | `config.py` の `SLOW_MO_MS = 800`〜`3000` にすると1操作ごとに待つ（既定は 0 = 全速力） |
| ポート8765が使用中 | `config.py` の `DUMMY_SERVER_PORT` を変更 |

## 動作確認済みの内容

ダミーサイトに対して、以下を実際に実行して確認済みです。

- 一連のフロー（アクセス → 遷移 → 入力 → 添付 → 送信 → 結果確認）が最後まで動作すること
- UIの処理フロー欄が各ステップの成功・スキップ・失敗を正しく表示し、クリックで
  実施内容のログと証跡スクリーンショットを確認できること
- 添付ファイルの中身が実際にサーバーへ到達していること（SHA256ハッシュの一致を確認）
- 非表示(`display:none`)の `input[type=file]` に対しても添付できること
- `<select>` を value属性・表示ラベルのどちらでも選択できること
- 以下のエラーが、それぞれ適切なメッセージでログに出力されること
  - 入力検証（氏名が空／日付形式が不正／添付ファイルが存在しない）
  - ポータルURLが404
  - セレクタ誤り（ポータルのリンク／入力欄／添付input／送信ボタン）
  - プルダウンに存在しない選択肢
  - CDP接続の失敗
  - サーバー側が申請を却下したケース
  - `DRY_RUN` による送信直前での停止
